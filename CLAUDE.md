## What This Is

An **example App** built with `@mctx-ai/app` framework. Demonstrates all framework capabilities in a single, well-commented reference implementation (`src/index.ts`).

**Purpose:** Serve as learning material and template for developers building Apps with this framework.

**Key Features:**
- Four tool patterns (sync string, object return, generators with progress, LLM sampling)
- Two resource patterns (static URI, dynamic URI template)
- Two prompt patterns (single-message, multi-message conversation)
- Tool annotation hints (readOnlyHint, destructiveHint, openWorldHint, idempotentHint)
- Structured logging throughout
- Environment variable configuration demo
- Comprehensive test coverage

## Common Development Commands

### Build
```bash
npm run build
```
Bundles `src/index.ts` ‚Üí `dist/index.js` using esbuild (minified ESM output).

Build output is gitignored on `main` ‚Äî the release pipeline commits `dist/index.js` to the `release` branch automatically.

### Development
```bash
npm run dev
GREETING="Howdy" npm run dev   # with env vars
```
Runs esbuild watch + mctx-dev hot-reload in parallel.

### Testing
```bash
npm test                        # run all tests
npm test -- --watch             # watch mode
npm test -- -t "greet"         # filter by name
```

Tests use Vitest. Each test creates a `Request` object, calls `server.fetch()`, and validates the JSON-RPC response.

### Linting and Formatting
```bash
npm run lint          # ESLint check
npm run format        # Prettier format
npm run format:check  # check only
```

## Architecture

### High-Level Structure

```
src/index.ts        ‚Üí Server implementation (all features in one file)
  ‚îú‚îÄ Tools          ‚Üí Functions LLM clients can invoke
  ‚îú‚îÄ Resources      ‚Üí Data exposed via URIs
  ‚îú‚îÄ Prompts        ‚Üí Reusable message templates
  ‚îî‚îÄ Export         ‚Üí Fetch handler for JSON-RPC 2.0 over HTTP

src/index.test.ts   ‚Üí Comprehensive tests for all capabilities
dist/index.js       ‚Üí Bundled output (generated by esbuild)
```

### Framework Pattern

**Registration:** All capabilities (tools, resources, prompts) follow a decorator pattern:
1. Define handler function with specific signature
2. Attach metadata (`.description`, `.input`, `.mimeType`)
3. Register with server via `server.tool()`, `server.resource()`, or `server.prompt()`

**Handler signatures:**
- **ToolHandler:** `(args, ask?, ctx?) => string | object | Promise<string | object>` ‚Äî attach `.description`, `.input`, `.annotations`
- **GeneratorToolHandler:** `function* (args, ctx?) { yield progress; return result; }` ‚Äî attach `.description`, `.input`, `.annotations`
- **ResourceHandler:** `(params, ctx?) => string` ‚Äî attach `.description`, `.mimeType`
- **PromptHandler:** `(args, ctx?) => string | conversation(...)` ‚Äî attach `.description`, `.input`

**The `ctx` parameter** (optional) provides access to authenticated user identity and channel event emission ‚Äî see `src/index.ts` for full usage examples.

**Schema system:** The `T` namespace provides type-safe schema builders (`.string()`, `.number()`, `.boolean()`, etc.) with validation constraints.

### Tool Annotations

MCP clients use annotation hints to decide how to present permission prompts and safety warnings to end users. Set them on any `ToolHandler` or `GeneratorToolHandler` via the `.annotations` property after defining the handler function.

**The four hints:**

| Hint | Type | Default | Meaning |
|---|---|---|---|
| `readOnlyHint` | `boolean` | `false` | Tool only reads data ‚Äî no state changes. Clients may suppress or soften the permission prompt. |
| `destructiveHint` | `boolean` | `true` | Tool may destroy or permanently alter data. Clients may show a stronger warning. |
| `openWorldHint` | `boolean` | `true` | Tool reaches outside the server (network requests, filesystem, external APIs). Clients may flag external access. |
| `idempotentHint` | `boolean` | `false` | Calling the tool multiple times with the same arguments produces no additional side effects after the first call. Clients may safely retry. |

Defaults are pessimistic ‚Äî if you don't set them, clients assume worst case (destructive, non-idempotent, open-world). Always set annotations explicitly.

`destructiveHint` and `idempotentHint` are only semantically meaningful when `readOnlyHint` is `false`. For read-only tools, omit them or expect clients to ignore them.

Hints are advisory ‚Äî they inform the client UI but do not enforce behavior in the server.

**Decision checklist ‚Äî choose the right profile:**

1. Does the tool write, delete, or modify anything? If no ‚Üí set `readOnlyHint: true`.
2. Is a write irreversible or data-destroying? If yes ‚Üí set `destructiveHint: true`.
3. Does the tool call an external service, make a network request, or read from the filesystem? If yes ‚Üí set `openWorldHint: true`.
4. Can the tool be safely retried with identical arguments? If yes ‚Üí set `idempotentHint: true`.

See `src/index.ts` for full annotation examples.

### Entry Point

`createServer()` initializes an App instance. The returned `server.fetch` property is a Web Standard Fetch API handler compatible with:
- Cloudflare Workers
- Deno Deploy
- Node.js with adapters
- mctx hosting platform

The handler processes JSON-RPC 2.0 requests over HTTP.

## Deployment Model

**mctx does not run build commands.** It serves whatever `dist/index.js` exists on the `release` branch ‚Äî it never executes `npm run build` or any equivalent at deploy time.

**`dist/` is gitignored on `main`.** Developers never commit build output. The release pipeline (see [Release Pipeline](#release-pipeline)) builds `dist/index.js` from source and commits it to the `release` branch automatically. This keeps `main` clean ‚Äî only source code lives there.

**Deployment trigger:** mctx deploys from the `release` branch. It watches for version changes in `package.json` on that branch. When the `version` field changes on a push to `release`, mctx starts a new deployment automatically. A push with no version bump does not trigger deployment, even if `dist/index.js` changed.

**Constraints:** Do not edit the `release` branch directly ‚Äî it is fully managed by CI. Do not commit `dist/index.js` on `main` ‚Äî the `.gitignore` excludes it. Do not expect mctx to run `npm install`, `npm run build`, or any other build step at deploy time.

## Release Pipeline

**Dual-branch model:** `main` (development) ‚Üí `release` (production). The `release` branch is fully managed by CI ‚Äî never edit it directly. All changes flow through `main` via pull request.

Pushing to `main` triggers `release.yml` ‚Äî the only path where `dist/index.js` gets committed to `release` and a new version is tagged.

### Conventional Commits

Version bumps are determined automatically from the commit message on `main`:

| Commit prefix | Version bump | Example |
|---------------|-------------|---------|
| `feat!:` or `fix!:` | **Major** | `feat!: redesign tool API` |
| `feat:` | **Minor** | `feat: add new resource` |
| Everything else | **Patch** | `fix: handle edge case`, `chore: update deps` |

**This repo uses squash merging.** PR title becomes the commit subject, PR description becomes the commit body. Therefore, **PR titles must follow conventional commit format** ‚Äî enforced by the `pr-title.yml` workflow.

## Three-Audience Model

Three artifacts serve different audiences:

- **`instructions`** (in `createServer()` or `package.json`) ‚Äî sent to AI clients (Claude, Cursor). Help the AI route to this App's tools. Name every tool, every resource URI pattern; be explicit. Code via `createServer()` takes precedence over `package.json`.
- **`README.md`** ‚Äî subscriber-facing product page on mctx. Write for humans: value proposition, use cases, examples.
- **`description` in `package.json`** ‚Äî one-liner shown on the mctx info page and MCP Community Registry. Be informative and scannable.

## package.json Fields

**Required fields:**

| Field | Purpose |
|-------|---------|
| `name` | App display name shown in the mctx dashboard and sent to AI clients as the App identifier |
| `version` | Triggers a new mctx deployment when it changes on a push |
| `description` | Shown on the App info page and MCP Community Registry |
| `main` | Path to the built JS file mctx serves (`dist/index.js`) |

**Important type field:**

| Field | Value | Purpose |
|-------|-------|---------|
| `type` | `"module"` | Required for ESM ‚Äî the framework uses ES module syntax |

**Optional fields:**

| Field | Purpose |
|-------|---------|
| `instructions` | AI client guidance ‚Äî alternative to setting it in code via `createServer()` (see [Three-Audience Model](#three-audience-model) for precedence rule). |

## Testing Patterns

### JSON-RPC Helper Functions

**`createRequest(method, params)`** ‚Äî Creates a `Request` object with proper JSON-RPC 2.0 structure. See `src/index.test.ts` for usage.

**`getResponse(response)`** ‚Äî Parses JSON-RPC response from `Response` object.

### Test Structure

Tests organized by capability type:
- **Tools** ‚Äî Invoke via `tools/call`, validate `result.content[0].text`
- **Resources** ‚Äî Read via `resources/read`, validate `result.contents[0].text` and `mimeType`
- **Prompts** ‚Äî Get via `prompts/get`, validate `result.messages[]` array structure
- **Server Capabilities** ‚Äî List via `tools/list`, `resources/list`, `prompts/list`

Error handling tests verify `result.isError === true` and error message content.

## CI Checks on Pull Requests

PRs to `main` run several automated checks:

- **test.yml** ‚Äî `npm ci` + `npm test` (Node 22)
- **check-dist.yml** ‚Äî Rebuilds `dist/` from source to verify the build succeeds. Since `dist/` is gitignored on `main`, this check validates that source changes produce a clean build ‚Äî not that a committed `dist/index.js` matches. The actual `dist/index.js` commit happens on the `release` branch via the release pipeline.
- **pr-title.yml** ‚Äî Validates PR title follows conventional commit format
- **pr-comment.yml** ‚Äî Posts a comment explaining squash merge behavior and version bump rules
- **dependabot-auto-merge.yml** ‚Äî Auto-merges Dependabot PRs

All GitHub Actions must be SHA-pinned (enforced by CI).
